## 1. Overview

OpenMath v2.0 replaces the planned React + Laravel stack with **Angular 18 + FastAPI**, sharing the same PostgreSQL 16 database that the existing Nuxt 4 app uses.

| Layer      | Technology                            | Port  |
|------------|---------------------------------------|-------|
| Frontend   | Angular 18.2 + PrimeNG 17 + PrimeFlex| 4200  |
| Backend    | FastAPI + asyncpg + Pydantic v2       | 8000  |
| Database   | PostgreSQL 16 (shared with Nuxt)      | 5432  |
| Admin      | Adminer                               | 8080  |

Both the Nuxt and Angular stacks can run **simultaneously** against the same database — the dev.ps1 launcher manages them as separate processes.

## 2. Architecture

```
┌───────────────┐        ┌───────────────┐
│ Angular SPA   │  HTTP  │  FastAPI       │
│ :4200         │ ─────► │  :8000/api/*   │
│ PrimeNG       │        │  asyncpg       │
└───────────────┘        └───────┬───────┘
                                 │
                    ┌────────────▼────────────┐
                    │  PostgreSQL 16           │
                    │  :5432 (Docker)          │
                    │  JSONB + GIN indexes     │
                    └─────────────────────────┘
```

### Key Architectural Choices

- **No ORM** — FastAPI uses raw SQL via asyncpg for maximum control and performance
- **Pydantic v2** schemas with `model_config = ConfigDict(from_attributes=True)` for clean serialization
- **Angular signals** (`signal()`, `computed()`) for reactive state, no NgRx or RxJS-heavy stores
- **Standalone components** — no NgModules, tree-shakeable
- **Lazy-loaded routes** — each feature page loaded on demand
- **New control flow** — `@if`, `@for`, `@switch` (Angular 17+ syntax, not `*ngIf`)

## 4. FastAPI Backend

### 4.1 Config (`config.py`)

Uses `pydantic-settings` to load from `../.env`:

```python
class Settings(BaseSettings):
    database_url: str = "postgresql://openmath:openmath@localhost:5432/openmath"
    cors_origins: list[str] = ["http://localhost:4200", "http://localhost:3000"]
    debug: bool = True
```

The `.env` file is shared with the existing Nuxt stack.

### 4.2 Database Pool (`database.py`)

A single asyncpg connection pool created via FastAPI's lifespan hook:

```python
@asynccontextmanager
async def lifespan(app: FastAPI):
    pool = await asyncpg.create_pool(dsn=settings.database_url, min_size=2, max_size=10)
    _state["pool"] = pool
    yield
    await pool.close()
```

All query functions call `get_pool()` → `pool.acquire()` for connections.

### 4.3 Queries (`queries.py`)

Raw SQL via asyncpg. Every function is async and uses `$1`, `$2` parameter binding. Results are converted to dicts via `_row_to_dict()` which handles UUID→str and datetime→ISO serialization. JSONB columns are parsed via `json.loads()`.

Key functions:
- `list_quiz_types()` — includes new `answer_type`, `template_kind` columns
- `create_session()` + `insert_questions()` — transactional session creation
- `submit_answer()` — stores both JSONB `response` and legacy `value` columns
- `get_session_by_id()` — joins session + questions + answers, grouped
- `get_student_performance_stats()` — per-student aggregate (correct/total by quiz type)
- `delete_all_schema_data()` — cascading truncate for admin reset

### 4.4 Services

| Service | Purpose |
|---------|---------|
| `difficulty.py` | Maps difficulty labels (easy/medium/hard) to multiplication table sets |
| `generator.py` | Creates N questions for a quiz type + difficulty, produces JSONB `prompt` payloads |
| `grader.py` | Grades an answer — supports `int`, `choice`, `tuple`, `fraction` answer types |
| `scoring.py` | Calculates percentage score with 2 decimal precision |
| `stats.py` | Aggregates performance data into buckets (overall + by quiz type) |

### 4.5 Routers

Each router is mounted under `/api` prefix in `main.py`:

```python
app.include_router(quiz_types_router, prefix="/api")
app.include_router(students_router, prefix="/api")
app.include_router(sessions_router, prefix="/api")
app.include_router(answers_router, prefix="/api")
app.include_router(stats_router, prefix="/api")
```

## 6. Database Migration

**File**: `db/migrations/0005_jsonb_modernization.sql`

### What it does

1. **Adds columns** to `quiz_types`: `answer_type` (default `'int'`), `template_kind` (default `'axb'`)
2. **Adds JSONB column** to `questions`: `prompt` (nullable)
3. **Adds columns** to `answers`: `response` (JSONB, nullable), `raw_input` (text, nullable)
4. **Backfills** existing data — creates JSONB prompt objects from legacy `operand_a`, `operand_b`, `operator` columns
5. **Creates GIN indexes** on `questions.prompt` and `answers.response` for fast JSONB queries
6. **Creates functional index** on `(questions.prompt->>'template')` for template-based lookups
7. **Creates index** `idx_sessions_student` on `sessions(student_id)` for student performance queries

### Idempotency

All statements use `IF NOT EXISTS` or `COALESCE` guards — safe to run multiple times.

### Backward Compatibility

All new columns are nullable or have defaults. Existing Nuxt app continues working unchanged.

## 8. How to Run

### Prerequisites

- Node.js 18+ and pnpm
- Python 3.11+ with venv at `.venv`
- Docker Desktop (for PostgreSQL)

### Quick Start (Full v2.0 Stack)

```powershell
# Option A: Use dev.ps1 interactive menu
.\dev.ps1
# Select item 21 (v2.0: Full Dev Up)

# Option B: Direct mode
.\dev.ps1 -Mode up-v2
```

### Step-by-Step Manual

```powershell
# 1. Start database
docker compose up -d

# 2. Apply migrations (including JSONB modernization)
.\scripts\apply-migrations.ps1

# 3. Create Python venv (if not exists)
python -m venv .venv

# 4. Install FastAPI dependencies
.\.venv\Scripts\python.exe -m pip install -r python-api\requirements.txt

# 5. Start FastAPI server
cd python-api
..\.venv\Scripts\python.exe -m uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload
cd ..

# 6. Install Angular dependencies
cd angular-app
pnpm install
cd ..

# 7. Start Angular dev server
cd angular-app
pnpm start
cd ..
```

## 9. API Reference

All endpoints are prefixed with `/api`.

### Quiz Types

| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/quiz-types` | List all quiz types (with answer_type, template_kind) |

### Students

| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/students` | List all students |
| GET | `/api/students/{id}` | Get student profile with performance stats |
| PATCH | `/api/students/{id}` | Update student display_name and/or learned_timetables |

### Sessions

| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/sessions?studentId=X` | List sessions (optionally filtered by student) |
| GET | `/api/sessions/{id}` | Get session detail with all questions and answers |
| POST | `/api/sessions` | Create new quiz session (generates questions) |

**POST `/api/sessions` request body:**
```json
{
  "quizTypeId": "uuid",
  "difficulty": "easy|medium|hard",
  "totalQuestions": 10,
  "studentId": "uuid",
  "studentDisplayName": "Name",
  "learnedTimetables": [2, 3, 5]
}
```

### Answers

| Method | Path | Description |
|--------|------|-------------|
| POST | `/api/answers` | Submit an answer to a question |

**POST `/api/answers` request body:**
```json
{
  "questionId": "uuid",
  "sessionId": "uuid",
  "value": 42,
  "response": { "answer": 42 },
  "rawInput": "42"
}
```

### Stats

| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/stats` | Database statistics (row counts) |
| GET | `/api/stats/{table}` | Browse rows from a specific table |
| POST | `/api/stats/reset` | Delete all data (with confirmation) |

## 10. Key Design Decisions

### Why asyncpg over SQLAlchemy?

The Nuxt app uses Drizzle ORM. We chose raw asyncpg to:
- Match the spec's emphasis on raw SQL
- Avoid adding another ORM layer
- Keep full control over query optimization (especially JSONB)
- Minimize dependencies

### Why signals over NgRx?

For an app of this scope, Angular signals provide sufficient reactivity:
- `signal()` for mutable state
- `computed()` for derived state
- No boilerplate (actions, reducers, effects)
- Framework-native (no external dependency)

### JSONB Dual-Write Strategy

Both legacy scalar columns and new JSONB columns are populated:
- `questions.operand_a/operand_b/operator` (legacy) + `questions.prompt` (JSONB)
- `answers.value` (legacy int) + `answers.response` (JSONB) + `answers.raw_input` (text)

This ensures **backward compatibility** with the Nuxt app while enabling richer data models for future quiz types (multi-step, fraction, choice).

### Standalone Components

Every Angular component is `standalone: true` — no NgModules:
- Better tree-shaking
- Clearer dependency graph per component
- Angular team's recommended pattern for Angular 17+
